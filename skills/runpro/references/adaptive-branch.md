# Adaptive Branch

Use this branch when the project does not cleanly fit document, presentation, code, data, or mixed patterns.

This is the catch-all route for novel, custom, or weak-signal projects.

## Default Rule

Do not force the project into an existing branch when the evidence is weak.

Instead:

1. identify the real end state
2. define the smallest viable handoff
3. choose a provisional execution path
4. validate against the actual project requirement, not against a borrowed template

## First Working Version

The first working version should prove that the project is moving toward the actual requested handoff, even if the artifact shape is unusual.

Examples:

- a custom package of files
- a hybrid handoff note plus assets
- a workflow output that does not map cleanly to one standard branch

## Validation Focus

- the real user-facing requirement is represented
- the chosen path is still practical
- non-standard outputs are organized for handoff
- completion is judged against the end state, not just the routing heuristic

## Common Repairs

- wrong initial route -> switch routes early and preserve completed useful work
- unusual output shape -> define a custom acceptance checklist in `execution-plan.md`
- ambiguous artifact -> state the provisional contract in `project-brief.md` and continue
