# Project State Files

Maintain these files inside the project root unless there is already a better equivalent.

By default, all runpro-generated files live under `runpro_workspace/`.

Keep the clean handoff folder at `runpro_workspace/submission/` so submission files stay top-level and immediately visible inside the generated workspace.

`runpro_workspace/submission/` should contain only the actual final submission files.

Rule:

- keep drafts, markdown working files, template reference files, style exemplars, audits, notes, helper exports, and duplicate variants outside `runpro_workspace/submission/`
- if the final deliverable is a written artifact and no explicit final format is stated, default the submission artifact to `.docx`

## 3A. `runpro_workspace/10_analysis/requirement-ledger.md`

Purpose:

- track each explicit requirement, prohibition, and formatting rule found in the project files
- preserve source traceability instead of collapsing multiple requirements into one summary
- make final requirement coverage auditable item by item

Suggested structure:

```text
# Requirement Ledger

## Requirement R1
### Source
### Type
### Requirement
### Status
### Satisfaction Evidence
### Notes
```

Recommended status values during execution:

- `pending`
- `satisfied`
- `blocked`
- `not applicable`

Rule:

- create one block per explicit requirement-bearing item, not one block per file
- include the source file, screenshot, or conversation reference for each requirement
- include content requirements, structure requirements, format requirements, prohibitions, and special emphasis rules
- do not rely on the project brief or requirement lock alone when the source files contain more specific requirements
- before handoff, every ledger item must be `satisfied`, `blocked`, or `not applicable`

## 3B. `runpro_workspace/10_analysis/rubric-ledger.md`

Purpose:

- convert rubric, brief, marking criteria, teacher feedback, required questions, required sub-parts, and format criteria into auditable grading checks
- drive academic writing from the grading standard rather than from a generic outline
- prevent broad summary language from hiding partial rubric coverage

Suggested structure:

```text
# Rubric Ledger

## Rubric Item C1
### Source
### Criterion
### Weight Or Importance
### Expected Evidence
### Draft Location
### Status
### Evidence
### Repair Action
### Notes
```

Recommended status values:

- `pending`
- `pass`
- `partial`
- `fail`
- `not assessable`

Rule:

- create one item per rubric criterion, required question, required sub-part, actionable teacher-feedback point, or format criterion
- include draft location evidence before claiming the criterion is passed
- do not call a central or high-weight item passed unless the answer is visible and direct
- if no rubric exists, create a `rubric-inferred` ledger and record the limitation
- before high-score handoff, central items should be `pass` unless a true blocker is documented

## 3C. `runpro_workspace/10_analysis/feasibility-check.md`

Purpose:

- determine whether the current folder contents are sufficient to complete the project
- distinguish recoverable gaps from blockers that need user or external input
- make the go or no-go decision explicit before execution starts

Suggested structure:

```text
# Feasibility Check

## Completion Verdict
## Execution Mode Recommendation
## High-Achievement Reachability
## Basis For Verdict
## Required Inputs Already Present
## Recoverable Gaps
## Missing Required Inputs
## Needed From User Or External Source
## Effect On Execution
```

Allowed verdicts:

- `Can Complete`
- `Can Complete With Recoverable Gaps`
- `Cannot Complete Yet`

Rule:

- do not claim `Can Complete` if essential assignment instructions, required templates, core source material, required data, or mandatory forms are still missing
- if the task is high-risk, state that `strict mode` is required instead of leaving execution mode implicit
- do not claim the `90+` target is realistically reachable if the available materials are too weak to support it
- do not treat a partial solution as enough if the assignment requires all parts to be completed
- missing submitter identity fields such as student name, student ID, candidate number, class, section, email, or phone number are not blockers; leave them blank by default and do not request them as standalone missing inputs
- if the verdict is `Cannot Complete Yet`, list exactly what must be added before execution can truthfully continue

## 5A. `runpro_workspace/10_analysis/final-audit.md`

Purpose:

- record a detailed end-of-project self-check
- make completion decisions evidence-based instead of impression-based
- force unresolved items back into execution before handoff

Suggested structure:

```text
# Final Audit

## Audit Scope
## Target Threshold
## Internal Score
## Estimated Score Band
## Strict-Mode Validation Chain
## Final Format Check
## Rubric Compliance Audit
## Academic Standards Audit
## Source-Claim Integrity Audit
## Academic Quality Audit
## Citation Micro-Audit
## Presentation Source Audit
## Requirement-By-Requirement Verdicts
- [pending] R1
## Validation Evidence
## Incomplete Or Failed Items
## Required Remediation Actions
## Blocking Issues
## Completion Decision
```

Rule:

- every hard requirement should appear in this audit with a pass, fail, incomplete, or blocked verdict
- every explicit requirement in `requirement-ledger.md` should be evaluated explicitly in the audit rather than implied by summary language
- the audit should compute a requirement-weighted internal score out of `100`
- the audit should state whether the current submission plausibly sits in the `90+` band
- the `Estimated Score Band` should record the most conservative credible current band rather than the most optimistic scenario
- for `strict mode` projects, the audit should record whether the validator-backed chain was completed cleanly
- when the assignment specifies formatting rules, the audit should record the final whole-document format-check result explicitly
- for graded academic written submissions, the audit should record criterion-by-criterion rubric compliance, including pass/partial/fail/not-assessable status and evidence for central criteria
- for assignment-style document work, the audit should record academic standards compliance, including citation style, in-text/reference matching, reference metadata, formatting, figure/table visual quality, visual balance, label readability, paragraph logic, headings, word count, appendices, and academic register where relevant
- for source-backed written submissions, the audit should record source-claim integrity for major factual, empirical, evaluative, and comparative claims
- for academic or source-backed graded written submissions, the audit should record prompt/rubric alignment, section-level analytical purpose, evidence quality, comparative synthesis, limitations or counterarguments, visual/table integration, citation-to-outcome specificity, empirical precision, theory/framework application, conclusion discipline, and citation support integrity when those checks are relevant
- for assignment-style document work, the audit should record the final citation micro-audit result explicitly, including any repaired non-blocking defects
- for source-backed presentation work, the audit should record the final presentation source audit result explicitly, including whether the deck references/source slides match the verified `source-log.md`
- for visually polished PPT/PPTX work using `$pptpro`, the audit should record PPTPRO-level production evidence: slide recipe map, visual enrichment/image plan, generated-image handling with the current latest built-in Codex/ChatGPT image tool where used, rounded-corner consistency, human-facing source text with no internal workflow language, original data/source figure preservation, rendered preview inspection, strict audit result, and any revised-range/full-deck assembly distinction
- if any fixable item is fail or incomplete, do not hand off yet
- if the submission is still below the `90+` target band, record why and what repairs are still required
- if a later reassessment lowers the credible band below the target, update `Estimated Score Band`, reopen `Required Remediation Actions`, and resume execution instead of defending the earlier ready state
- handoff can proceed only when all fixable items pass or a true blocker remains

## 8A. `runpro_workspace/10_analysis/source-claim-audit.md`

Purpose:

- verify that major claims are actually supported by the cited sources
- prevent decorative, mismatched, overbroad, or invented-looking citations
- make source-backed writing auditable before final handoff

Suggested structure:

```text
# Source-Claim Integrity Audit

## Claim C1
### Draft Location
### Claim
### Claim Type
### Source Support
### Support Verdict
### Evidence Boundary
### Required Repair
### Notes
```

Recommended support verdict values:

- `supported`
- `needs narrowing`
- `needs source`
- `unsupported`
- `not assessable`

Rule:

- map major factual, empirical, evaluative, and comparative claims to specific source support
- preserve measured outcome, direction, magnitude, units, sample, method, population, date range, and conditions when they affect interpretation
- do not assume a citation supports a claim if the source text cannot be checked
- repair, narrow, cite, or remove unsupported claims before high-score handoff

## 9. `runpro_workspace/10_analysis/approval-gate.md`

Purpose:

- create a strict pre-execution requirement lock
- capture the user-approved scope before execution
- prevent work from continuing on an unconfirmed or drifting interpretation

Suggested structure:

```text
# Requirement Lock

## Project Goal
## Final Deliverable
## Execution Mode
## Expected Genre And Register
## Expected Depth Or Quality Tier
## Target Quality Threshold
## All Required Parts Must Be Completed
## Required Structure Or Sections
## Required Format Or Template
## Explicit Formatting Requirements
## Default Formatting Fallback
## Non-Negotiable Requirements
## Explicitly Forbidden Moves
## Accepted Inferences
## Personal Identity Fields
## Open Questions Needing Confirmation
## Locked Execution Direction
## Approval Status
## User Confirmation Record
```

Default status before user reply: `Pending user confirmation`

Rule:

- do not begin execution until this file is approved by the user
- derive this lock from `requirement-ledger.md`, not from memory alone
- if the project is in `strict mode`, record that explicitly here instead of assuming it from context
- if execution later needs a different structure, template, or substitute path, return here and re-lock with the user first
- if the assignment does specify formatting rules, record them explicitly here instead of assuming they will be remembered later
- if the assignment does not specify font or layout rules, record the chosen standard academic fallback here instead of leaving formatting implicit
- if the assignment does not specify report-table or process-diagram styling, record the default fallback here: black-and-white three-line tables for report tables and black-and-white styling for inserted project flowcharts or process diagrams
- if the template contains blank submitter identity fields such as student name or student ID, record that they will be left blank by default and do not ask for confirmation solely to fill them
- if a required template or sample implies a formal report style, record that quality bar here instead of silently downgrading the document type
