# Collaboration Framework

Use this as a reasoning reference for the skill, not as a rigid user-facing ritual.

The optimized system should absorb this logic while reducing the amount of manual setup the user has to perform.

## 1. Define The End State

Clarify:

- project goal
- final deliverable
- deadline
- evaluation standard

If these are missing, infer conservatively and mark the uncertainty.

## 2. Collect Existing Materials

Gather available:

- files
- screenshots
- links
- templates
- prior versions
- rule notes

Inventory first. Do not jump straight into production.

## 3. Verify Before Acting

Separate:

- confirmed facts
- exact assignment instructions
- required structure, section order, or methodology emphasis
- required template, form, or delivery shell
- explicitly forbidden moves
- assumptions
- tool limitations
- protected files or systems

Execution quality depends on boundary clarity.

## 4. Perform Gap Analysis

Classify the project into:

- already available
- missing but recoverable
- risky or constrained

This determines whether the main path is safe or whether fallback planning is required.

## 5. Decide Whether The Current Materials Are Enough

Before execution, answer this explicitly:

- can the project be completed with the current folder contents
- what required inputs are present
- what required inputs are missing
- which gaps are recoverable by the workflow itself
- which gaps need the user or an external source

Do not confuse "enough to start" with "enough to finish." The question here is whether the project can be completed to the required standard.

## 6. Choose An Execution Path

Always define:

- primary path
- backup path
- contingency path that still satisfies all locked requirements

Do not treat one fragile route as the only route unless it truly is.

## 7. Build A Runnable First Version

First version goal:

- prove the structure
- prove the core output
- prove the critical path works

Do not try to polish everything before basic viability is established.

## 8. Validate Continuously

After each meaningful step, check:

- content correctness
- structure correctness
- requirement alignment
- compatibility or presentation issues

Catch defects early.

## 9. Iterate From Usable To Shippable

Think in passes:

1. exists
2. correct
3. polished

This prevents early overinvestment in details before the fundamentals are stable.

## 10. Close Delivery Properly

Before handoff, organize:

- main output
- support files
- notes or explanation
- evidence if needed
- names and versions

Completion is not just the artifact. It is the artifact in a usable delivery state.

## 11. Preserve Reusable Method

After completion, capture:

- startup checklist
- validation checklist
- common issue handling
- final delivery pattern

The project should improve the next project.

## Communication Pattern

If the user does provide startup fields, these are the most useful ones:

- project name
- project goal
- final deliverable
- deadline
- evaluation standard
- existing materials
- constraints
- current biggest problem
- what to do first

Optimized assistant sequence:

1. bootstrap the project folder
2. read priority materials
3. reconstruct the exact requirements
4. determine boundaries
5. decide whether the current materials are sufficient to complete the work
6. lock a feasible path
7. ask the user to confirm the requirement lock
8. build skeleton
9. build usable version
10. optimize only where it matters
11. prepare delivery version

## Requirement Lock Rule

After requirements, goals, deliverables, route, required structure, template bindings, and must-not-do items are reconstructed, pause and ask the user to confirm that requirement lock.

Write this user-facing confirmation request in Chinese by default. The message should summarize the project goal, final deliverable, locked requirements, execution plan, blockers or open questions, and end with a direct request for confirmation or correction. Internal state files may remain in the existing working language, but the review-and-confirm response shown to the user should not be English unless the user explicitly requests English.

Do not continue into execution until the user confirms or corrects it.

Once confirmed, treat those details as binding.

Do not silently:

- change the section structure
- change the required emphasis
- substitute a different form or template
- reuse a draft the user excluded

If the exact required form, template, or shell is missing, surface the blocker and ask instead of inventing a replacement.

If the current folder still lacks essential materials needed to finish the project, say so clearly and list exactly what must be added.

After confirmation and execution, default to result-only output unless the user asks for more detail.

## Problem Escalation Format

When something goes wrong, respond in this structure:

- problem
- reason
- available options
- recommended path

Keep momentum unless a true blocker remains.
