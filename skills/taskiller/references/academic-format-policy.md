# Academic And Format Policy

Use this policy for essays, papers, literature reviews, source-backed reports, and source-backed presentations.

## Research-First Rule

For academic or source-backed tasks:

1. identify the assignment requirement
2. identify the required citation style
3. gather sufficient real sources
4. log them in `taskiller_workspace/10_analysis/source-log.md`
5. verify the source base
6. only then begin writing or slide synthesis

Do not write a literature-backed artifact first and search for sources later.

## Literature Selection Rule

For most academic and source-backed tasks, prioritize literature from roughly the last five years.

Default preference order:

1. recent sources from the last five years
2. higher-impact or field-recognized journals, major reviews, guidelines, and authoritative publishers
3. older sources only when they are seminal, foundational, method-defining, guideline-level, or explicitly required by the assignment

Do not build the source base mainly from older papers just because they are easier to find.

If an older source is kept, note why it is still necessary.

## Citation Style Rule

The citation style must come from the assignment, not from guesswork.

Examples:

- APA
- Harvard
- MLA
- Chicago
- Vancouver or numbered styles

If the style requirement is missing, mark it as unresolved and avoid inventing a style requirement.

## Assignment-Format Rule

If the assignment provides explicit formatting rules, treat them as binding from the start of drafting rather than as a cleanup step to defer until the end.

This includes items such as:

- font family
- font size
- spacing
- margins
- heading style
- title-page rules
- page numbering
- paragraph alignment
- citation presentation
- figure or table caption format

Lock these requirements in the requirement lock before full drafting begins whenever they are available.

## Rubric-First Writing Rule

For graded academic writing, source-backed reports, literature reviews, proposals, dissertations, or teacher-feedback-driven revisions, do not draft from a generic outline first.

Before full drafting:

- build `taskiller_workspace/10_analysis/rubric-ledger.md`
- split the rubric, assignment brief, required questions, required sub-parts, teacher feedback, and format criteria into separate ledger items
- record source, weight or importance, expected evidence, planned draft location, status, evidence, and repair action
- use statuses `pass`, `partial`, `fail`, or `not assessable`
- if no rubric exists, create a `rubric-inferred` ledger from the brief and genre and state the limitation

During drafting:

- each central rubric item should have an explicit section, paragraph, table, figure, appendix, or source-backed answer
- do not rely on a broad discussion to satisfy a specific question
- do not polish language before the core rubric structure is covered
- if a high-weight item cannot be answered from available materials, treat it as a blocker or recoverable gap rather than hiding it

Before handoff:

- every central rubric item should be `pass` or explicitly blocked
- multiple `partial` rubric items should prevent `90+ plausible` signoff
- `not assessable` is not acceptable for high-score readiness unless the missing assessment evidence is outside the project and documented as a true blocker

## Source-Claim Integrity Rule

For source-backed academic writing, do not treat citation presence as proof of source support.

Before and during drafting:

- create or update `taskiller_workspace/10_analysis/source-claim-audit.md`
- map major factual, empirical, evaluative, and comparative claims to source support
- check whether the cited source actually supports the specific claim attached to it
- preserve direction, magnitude, units, sample, method, population, date range, and conditions when they matter
- tie broad claims such as `better`, `effective`, `significant`, `sustainable`, `high quality`, or `improved` to measured or clearly defined outcomes
- identify uncited, overbroad, decorative, mismatched, invented-looking, or not-assessable citations
- repair, narrow, cite, or remove claims that cannot be supported truthfully

If the source text is unavailable, mark the source-claim check as limited or not assessable. Do not assume the citation supports the claim.

## Academic Standards Gate

Academic conventions are binding quality gates, not cosmetic polish.

Check these during drafting and again in the final file when relevant:

- citation style is identified from the assignment rather than guessed
- in-text citations and reference-list entries match each other
- reference-list ordering, punctuation, capitalization, italics, hanging indents, numbering, or superscripts match the required style when assessable
- source author names, initials, years, titles, journal/proceedings/book details, DOI/URL, and access details are not simplified or invented
- borrowed ideas, statistics, definitions, methods, frameworks, tables, figures, and distinctive claims are cited
- paragraphs follow academic reasoning: claim, evidence, analysis, and implication
- core terms, scope boundaries, comparison criteria, methodology, and framework choices are defined before evaluative claims use them
- headings, captions, tables, figures, appendices, page layout, word count, and file format follow the brief or template
- figures and tables are referred to in the text, captioned correctly, interpreted in prose, and visually balanced enough to look academic at final document size
- language is formal, precise, and field-appropriate without generic filler or unsupported exaggeration

## Personal Identity Fields

Cover-page, title-page, header, footer, or submission-form fields for personal identity are optional to fill by default.

Default rule:

- do not stop the workflow or ask the user only to fill fields such as student name, student ID, candidate number, class, section, email, phone number, or similar submitter identifiers
- do not invent or infer these values
- if these fields are blank in a template, leave them blank or preserve the empty placeholder
- if a source file already contains user-provided identity text, preserve that existing text unless the user asks to remove it
- missing identity fields should not make the document incomplete, blocked, or not ready for review

This exception applies only to submitter identity metadata. It does not apply to required content names, source author names, case names, institutional names, dates, participant labels, or ownership references that affect the academic content.

## Bold Emphasis Rule

For written deliverables, use restrained typography by default.

- Bold is allowed for the document title and the heading hierarchy.
- Do not use bold in body paragraphs, lists, captions, table body text, or other running text merely to create emphasis.
- If a template or assignment explicitly requires bold labels or another bold pattern inside body text, follow that requirement exactly and no broader.
- If a draft or exporter introduced bold emphasis in body text without an explicit requirement, remove it during the final formatting pass.

## Default Table And Flowchart Style

For written reports, use restrained black-and-white visuals by default unless the assignment, template, or sample clearly requires another style.

- Report tables should default to black-and-white three-line table styling.
- Use white cell backgrounds, black text, and the standard top rule, header-separator rule, and bottom rule as the default table structure.
- Do not default to colored header fills, zebra striping, gradient cards, or boxed full-grid borders unless the materials explicitly require them.
- For figures or charts embedded in essays, papers, theses, or report-style documents, do not place a duplicate title inside the chart image when the document will provide a figure caption or surrounding text already names the figure.
- Use the external caption, such as `Figure 1. ...`, as the figure title in written documents; keep necessary axis labels, legends, tick labels, units, and data annotations inside the figure.
- Only include an internal chart title when the figure is a standalone image, dashboard element, slide graphic, or the assignment/template explicitly requires the title to be inside the visual.
- Inserted project flowcharts or process diagrams should default to black-and-white only.
- Use black lines, arrows, and text on a white background, with white or unfilled nodes unless a different explicit visual convention is required.
- Make hierarchy through layout, spacing, labels, and line weight rather than through color coding when using this fallback.

## Academic Figure And Chart Quality

Figures, charts, and diagrams in essays, papers, theses, and report-style documents should look balanced, restrained, and academically polished.

Default visual rules:

- choose the chart or diagram type that makes the comparison easiest to read; do not use decorative 3D effects, shadows, gradients, or chart junk
- keep the plot, legend, labels, and whitespace visually balanced so the figure does not feel top-heavy, bottom-empty, left-stranded, or crowded on one side
- center or optically balance the data region within the figure canvas; avoid excessive blank margins caused by legends, long labels, or exported padding
- use consistent typography across figures, preferably the document's academic font family or a clean sans-serif that remains readable when embedded
- keep axis labels, legend labels, units, tick labels, and data labels readable at final document size
- use restrained line weights, marker sizes, gridlines, and borders; gridlines should support reading without dominating the data
- use grayscale or a limited, colorblind-safe palette with one or two functional accents when color is needed; avoid loud, decorative, or one-note saturated palettes
- place legends where they clarify the data without pulling the figure's visual center off balance; direct labels may be better when they reduce eye travel and clutter
- avoid overlapping text, clipped tick labels, clipped legends, tiny annotations, or dense data labels that make the figure look unfinished
- when the workflow allows, preserve editable or vector outputs such as SVG or PDF until final embedding; rasterize only when the final delivery format requires it
- keep scale choices honest and academically defensible; do not use misleading axes, inconsistent intervals, or cropped baselines unless the reason is clear and appropriate
- for multi-panel figures, align panel sizes, axes, label positions, and spacing so the set reads as one coherent figure
- for multi-panel figures, give each panel a distinct analytical role such as overview, comparison/deviation, validation, or relationship/mechanism; do not keep two panels that answer the same question unless the redundancy is clearly useful
- export or embed figures at sufficient resolution for the final `.docx` or PDF, and inspect the actual delivery file rather than only the source chart

If a chart looks mechanically exported, visually unbalanced, too decorative, or difficult to read at final size, repair it before handoff.

## Default Formatting Fallback

If the assignment materials do not specify font, font size, spacing, margin, alignment, or other layout rules, use a standard academic default format instead of leaving the document stylistically undefined.

Default fallback profile:

- font family: `Times New Roman` or the closest standard academic serif available in the target environment
- body font size: `12 pt`
- text color: black
- line spacing: `1.5`
- paragraph spacing before: `0`
- paragraph spacing after: `0.5` lines
- page margins: approximately `2.54 cm` / `1 inch` on all sides
- body alignment: left-aligned unless the target template or institution clearly expects justified text
- heading hierarchy: consistent and restrained, with no decorative colors or styling
- bold usage: title and headings only; no bold emphasis inside running body text unless explicitly required
- report table style: black-and-white three-line tables by default
- embedded figure/chart title style: no duplicate internal chart title when an external figure caption is used
- embedded figure/chart visual style: balanced, academic, readable at final size, with restrained color and no decorative chart effects
- inserted process-diagram style: black-and-white only by default

If the recipient or platform clearly uses a different institutional default, follow that instead.

When this fallback is being used, note it explicitly in the requirement lock or plan so the formatting decision is inspectable.

## Template-First Rule

If the assignment provides a real template, exemplar, or official shell, treat it as more than visual decoration.

The template may encode:

- expected report genre
- section rhythm and depth
- title and heading hierarchy
- paragraph density
- formatting conventions that affect how formal the submission feels

When such a template exists:

- extract its structure and style parameters before drafting the full body
- use it to infer the expected quality bar
- do not write a generic draft first and try to force the template on at the end unless no better path is available

If the template cannot be parsed or applied, surface that as a blocker or risk in the requirement lock and feasibility check.

## Register Interpretation Rule

Phrases like "not too academic" or "more like an analyst report" should be interpreted conservatively.

Default interpretation:

- reduce needless academic stiffness
- keep the work formal, structured, and submission-grade
- do not downgrade it into a short memo, casual note, or thin answer sheet unless the assignment clearly asks for that artifact

## Suggested Source Minimums

Use the assignment requirement first.

If the assignment does not state a number, use these defaults as a floor:

- essay or short paper: `5`
- literature review: `8` or more
- source-backed presentation: `3`
- research-backed report: `5`

If the task clearly needs broader coverage, raise the number instead of staying at the floor.

## Recency Preference

Unless the assignment or topic clearly requires historical coverage:

- prefer a source base where most citations come from the last five years
- treat older sources as exceptions, not the default
- prefer high-impact, core-journal, review, guideline, or otherwise field-recognized sources over low-signal sources

For typical literature-backed work, use `scripts/validate_source_log.py` with a recent-source check such as:

- `--preferred-recent-years 5`
- `--min-recent-ratio 0.6`

Raise the ratio when the task is highly current or fast-moving. Relax it only when the topic genuinely depends on older foundational literature.

## Document Formatting Rule

For essay or paper deliverables, the final delivery-format check should include:

- assignment-specific formatting requirements were applied during drafting rather than retrofitted only at the end
- in-text citations match the required style
- reference list matches the required style
- document genre, register, and depth still match the approved report type
- superscripts or numbered citations are correct when required
- document text color is consistently black unless the assignment explicitly says otherwise
- if no font or layout rule was provided, the document follows the default academic fallback format consistently
- if a real template exists, heading and body styling are aligned with that template before signoff
- bold styling is limited to the title and heading hierarchy unless the assignment or template explicitly requires bold in body text
- figures or charts render correctly in the actual document format
- figures or charts embedded in the document do not contain duplicate internal titles when an external figure caption/title is provided
- figures, charts, and diagrams are visually balanced, readable at final size, academically restrained, and free of clipped labels, crowded legends, decorative chart effects, or excessive blank canvas
- report tables use black-and-white three-line styling unless the assignment or template explicitly requires another table format
- inserted project flowcharts or process diagrams use black-and-white styling unless the assignment or template explicitly requires another visual rule
- tables, charts, and other visual elements have visible captions or titles when the genre expects them, and table header rows are visually distinguishable from body rows
- heading hierarchy and spacing are consistent

Scripted or programmatic document exporters must not silently introduce decorative heading colors, theme colors, or hyperlink-like text colors. Unless the assignment explicitly requires colored text, all visible document text, including headings, should render in black in the actual delivery file.

For business-report or analyst-report style documents, do not treat a table as complete merely because the data grid exists. In the actual delivery file, verify that:

- each meaningful table has a visible table label or title such as `Table 1. ...` when that genre convention is expected
- each table follows black-and-white three-line styling by default unless the source requirements clearly specify another style
- the first row reads clearly as a header row rather than as ordinary body text
- the surrounding prose makes the table feel analytically integrated rather than dropped in as a raw grid
- the table does not visually read as "unfinished" due to missing captioning, missing header emphasis, or unstyled export defaults

For inserted project flowcharts or process diagrams in report-style documents, verify that:

- the diagram uses black-and-white styling by default unless the source requirements clearly specify another style
- meaning is not carried only by color differences
- arrows, connectors, and labels remain readable when printed or viewed in grayscale

If a late formatting or exporter change is made to fix color, captions, headers, spacing, or similar document mechanics, rerun the page-structure check afterward. Do not assume title-page isolation, executive-summary pagination, page count, or table placement survived the late change unchanged.

After the full document content is complete, run one more whole-document formatting pass against the assignment requirements in the real delivery file whenever feasible.

If any formatting defect is found in this final pass, fix it and recheck before handoff.

## Academic Argumentation Quality Gate

For graded essays, academic reports, literature reviews, dissertation-style submissions, proposals, and source-backed assignments where analysis quality affects the mark, do not treat a document as high-scoring merely because it is complete, cited, and formatted. Run an argumentation-quality pass before final handoff.

Do not force this gate onto unrelated deliverables. For code projects, workflow notes, administrative documents, lightweight summaries, data artifacts, form filling, design-only presentations, or business deliverables whose rubric does not require academic argumentation, use the route-specific quality gate instead. If a project is mixed, apply this gate only to the academic written component.

This pass must check:

- **Rubric compliance:** every rubric criterion, required question, required sub-part, and actionable teacher-feedback point is separately assessed with status, draft location, evidence, and repair action.
- **Academic standards:** citation style, in-text/reference matching, reference-list correctness, formatting, figure/table visual quality, visual balance, paragraph reasoning, headings, word count, appendices, and academic register are checked against the brief or template.
- **Source-claim integrity:** major factual, empirical, evaluative, and comparative claims are mapped to source support, and unsupported, overbroad, mismatched, decorative, or not-assessable citations are repaired before signoff.
- **Prompt and rubric alignment:** the thesis, research question, or central judgement directly answers the assignment prompt. Every rubric criterion, required question, and required sub-part should have a visible answer in the body rather than being implied by the general topic.
- **Section-level analytical purpose:** every major section should have a clear job in the argument, such as defining criteria, comparing evidence, applying theory, interpreting data, testing feasibility, or deriving implications. Sections that only provide generic background should be condensed or converted into analysis.
- **Paragraph-level reasoning:** analytical paragraphs should connect claim, evidence, and implication. A paragraph that only describes a source and ends with a citation is not enough for high-score signoff when the assignment expects evaluation.
- **Evidence hierarchy:** high-stakes factual, scientific, technical, policy, or performance claims should rely on peer-reviewed, guideline, official, primary-data, or otherwise authoritative evidence when available. Lower-quality evidence may support context or examples, but should not carry the main proof burden.
- **Critical evaluation:** contrary findings, limitations, uncertainty, trade-offs, or boundary conditions should be acknowledged where the topic, rubric, or evidence base expects evaluation rather than one-sided advocacy.
- **Definitions and criteria:** key terms, scope boundaries, comparison criteria, framework choices, and methodology assumptions should be defined before they are used to make evaluative claims.
- **Visual integration:** every meaningful analytical table, figure, diagram, checklist, or model is explicitly interpreted in nearby prose. The prose should state what the visual shows, what trade-off or comparison it supports, and how it advances the report's judgement. Decorative visuals, administrative checklists, or required forms do not need this treatment unless they are used as evidence.
- **Comparative synthesis:** sources and examples are compared against common criteria instead of listed one by one. Where the topic includes alternative approaches, compare them by performance, evidence strength, limitations, feasibility, cost, risk, sensory/user effect, or other relevant dimensions.
- **Literature patterning:** literature coverage should identify patterns, disagreements, evidence strength, and gaps. Do not accept a source-by-source catalogue as synthesis when the assignment calls for a literature review or research-backed analysis.
- **Repetition control:** a central concept may be introduced clearly once, but later sections should develop it through evidence, cases, or contrasts. Repeated mechanism explanations should be shortened or replaced with analysis.
- **Citation-to-outcome specificity:** claims such as `better performance`, `improved stability`, `stronger effect`, or `higher quality` should identify the measured outcome where the cited source allows it, such as overrun, drainage, viscosity, elastic modulus, hardness, oral friction, sensory score, storage stability, accuracy, yield, time, cost, or field-specific endpoint.
- **Empirical precision:** numerical and empirical claims should preserve direction, magnitude, units, groups, experimental conditions, and comparison baselines when those details affect interpretation. Avoid turning measured results into vague adjectives.
- **Theory, method, or course-concept application:** when the assignment provides frameworks, theories, methods, models, or course concepts, the work should apply them explicitly rather than only mentioning them in passing.
- **Evidence boundary discipline:** commercial pages, product labels, marketing material, screenshots, and non-peer-reviewed examples may show market positioning or ingredient examples, but must not be used as proof of scientific performance, nutrition superiority, sensory quality, or sustainability unless they contain relevant measured evidence.
- **Table-driven conclusions:** at least the major tables should be followed or preceded by prose that extracts a conclusion from the table. A caption alone is not enough.
- **Argument flow:** topic sentences and transitions should make the reasoning traceable so the reader can tell why each section follows from the previous one and how it contributes to the final judgement.
- **Conclusion discipline:** the conclusion should give a final judgement, implications, and limits. It should be shorter and sharper than the main analysis and should not repeat the full body argument.
- **Citation support integrity:** every citation should correspond to a real source actually used, and the attached claim should be one the source can support. Decorative, mismatched, or authority-padding citations are not acceptable.
- **Register and precision:** obvious grammar defects, awkward machine-like phrasing, mistranslated idioms, and vague academic claims must be fixed before high-score signoff, even if the facts are broadly correct.

If this pass finds a fixable issue, the project is not `Ready to submit` for high-achievement purposes. Apply targeted repairs and rerun the relevant quality check.

## Citation Micro-Audit Rule

For assignment-style documents, do not stop after the broad style and layout check passes.

Run one more citation-and-reference micro-audit after the document already appears handoff-ready.

This micro-audit should explicitly check:

- in-text citations and the reference list are mutually consistent
- the reference list is sorted correctly for the required style
- author names, initials, and diacritics match the real source metadata rather than a simplified rewrite
- no-author, unpublished, classroom, or local materials are cited using the style-correct title-based or equivalent fallback rather than a filename-like shortcut
- no filename, folder name, or ad hoc label is masquerading as a proper source title in the citation system

Treat this as a mandatory final-quality pass for graded written work because these defects are often non-blocking but still grade-relevant.

If any micro-defect is found, fix it, recheck it, and only then hand off the document.

## Presentation Formatting Rule

For source-backed presentations, the final delivery-format check should include:

- citations or references appear in the required style or expected slide convention
- the end references or sources slide was generated from the verified `source-log.md`, not from unverified memory or mixed draft notes
- academic references are distinguished from datasets, websites, images, or other provenance entries unless the assignment explicitly requires one combined source list
- slides are readable and visually non-overlapping
- charts and figures render correctly in the actual presentation file
- visual hierarchy is clear
- the deck remains editable in the required format

For source-backed presentations, run one more source audit after the deck already appears handoff-ready.

This source audit should explicitly check:

- the latest verified `source-log.md` and the final deck references/source slides are consistent
- scholarly references are not mixed with data, media, website, or source-entry-point provenance in a way that falsely reads like a standard academic reference list
- if both types are needed, they are separated clearly by slide or labeled section such as `References` and `Data/Media Sources`

If any source-audit defect is found, fix the deck, recheck it, and only then hand off the presentation.

## Actual File Check Rule

The final check should happen in the real delivery format whenever feasible:

- `.docx` or equivalent for document deliverables
- `.pptx` or equivalent for presentation deliverables

If a written deliverable does not have an explicitly required final format, treat `.docx` as the default submission format and run the final check against that file.

Do not treat markdown working files as the clean final submission artifact unless the assignment explicitly requires markdown or plain text submission.

If native-format checking is not possible in the current environment, the limitation must be stated explicitly. Do not pretend that a native-format check happened if it did not.
